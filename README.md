# Football Prediction AI train AI

ဒီ `JOURNEY.md` ဖိုင်လေးက ပုံမှန် Documentation တွေလို အောင်မြင်မှုချည်းပဲ ကြွားထားတဲ့စာတမ်းမျိုးမဟုတ်ဘဲ၊ Football (ဘောလုံးပွဲ) ရလဒ်တွေကို ခန့်မှန်းနိုင်တဲ့ AI (Machine Learning Model) တစ်ခုကို အစကနေအဆုံး ဘယ်လိုတည်ဆောက်ခဲ့လဲဆိုတာကို အမှား၊ အမှန်၊ ကျရှုံးမှု အားလုံးကို ရိုးရိုးသားသား ချရေးထားတဲ့ နေ့စဉ်မှတ်တမ်း (Journal) တစ်ခုဖြစ်ပါတယ်။ 

ဒီထဲက အကြောင်းအရာတွေကို Programming သဘောတရားတွေ၊ တကယ့်လက်တွေ့ ဥပမာတွေနဲ့ ချိတ်ဆက်ပြီး အပိုင်းလိုက် အသေးစိတ် ရှင်းပြထားပါတယ်။

---

### အပိုင်း (၁) - အိုင်ဒီယာနဲ့ ပုန်းကွယ်နေတဲ့ ထောင်ချောက်များ (The Trap)

အခြေခံအိုင်ဒီယာက ရိုးရှင်းပါတယ်။ AI ကို Machine Learning Engineer တစ်ယောက်လို သဘောထားပြီး Model တွေဆောက်ခိုင်းမယ်၊ စမ်းသပ်ခိုင်းမယ်၊ အလုပ်ဖြစ်တာကို ယူမယ်။ ဒါပေမဲ့ ဒီလိုလုပ်တဲ့အခါ အကြီးမားဆုံး ထောင်ချောက်နှစ်ခု ရှိပါတယ်။

1. **Leakage (အဖြေကြိုသိနေခြင်း):** ဒါက Code ရေးတဲ့အခါ Variable တွေ မှားခေါ်သုံးမိသလိုပါပဲ။ ဥပမာ - ပွဲမစခင် ခန့်မှန်းရမယ့် Model ထဲကို ပွဲပြီးမှရမယ့် "အဝါကတ် အရေအတွက်" (သို့) "ဂိုးပေါက် ကန်သွင်းမှု" တွေကို Data အနေနဲ့ မှားထည့်မိတာမျိုးပါ။ ဒီအခါ Model က အဖြေကို ခိုးကြည့်ပြီး တွက်တဲ့အတွက် ရလဒ်တွေက 100% မှန်နေတတ်ပါတယ်။ တကယ့်လက်တွေ့မှာတော့ သုံးမရပါဘူး။
2. **Self-flattery (ကံကောင်းတာကို တော်တယ်လို့ ထင်ခြင်း):** Model ပုံစံ အခု ၂၀ လောက် ရမ်းစမ်းလိုက်တဲ့အခါ တိုက်ဆိုင်မှုကြောင့် တစ်ခုက အဖြေမှန်သွားတာကို Model ကောင်းသွားပြီလို့ ထင်ယောင်ထင်မှား ဖြစ်တတ်ပါတယ်။

ဒါတွေကို ကာကွယ်ဖို့အတွက် **စည်းမျဉ်း (Constitution)** တွေ အရင်ချခဲ့ပါတယ်။ အရေးကြီးဆုံးကတော့ "ပွဲမစခင် သိနိုင်တဲ့ Data (Knowable before kickoff) ကိုပဲ သုံးရမယ်" နဲ့ "အချိန်နောက်ပြန်သွားပြီး Train လုပ်ခွင့်မရှိဘူး (Chronological only)" ဆိုတဲ့ အချက်တွေပါပဲ။

---

### အပိုင်း (၂) - အုတ်မြစ်ချခြင်း (Data, Validation နဲ့ Baselines)

Model ကို အကဲဖြတ်ဖို့အတွက် Data တွေကို အပိုင်း ၃ ပိုင်း ခွဲလိုက်ပါတယ်။ 
* **Train Split (2000-2022):** Model ကို စာသင်ပေးမယ့် အတိတ်က Data များ။
* **Validation Split (2022-2024):** Model တွေကို အချင်းချင်း ယှဉ်ပြိုင်ခိုင်းပြီး အကောင်းဆုံးကို ရွေးချယ်မယ့် စာမေးပွဲ။
* **Test Split (2024-2026):** **(အရေးကြီးဆုံး)** ဒီ Data ကို လုံးဝ (လုံးဝ) သော့ခတ်ထားပါတယ်။ နောက်ဆုံး အကောင်းဆုံးဆိုတဲ့ Model ထွက်လာမှသာ ဒီအပေါ်မှာ တင်စမ်းမှာပါ။ (စာမေးပွဲ စစ်မေးခွန်း အစစ်ပါ)

**အကဲဖြတ်မယ့် စနစ် (Metrics):**
အဓိကကတော့ **Log Loss** လို့ခေါ်တဲ့ စနစ်ကို သုံးပါတယ်။ သူက "ဘယ်လောက် မှန်သလဲ" ဆိုတာထက် "မှားတဲ့အခါ ဘယ်လောက် ယုံကြည်မှုလွန်ကဲနေလဲ" ဆိုတာကို အပြစ်ပေးတဲ့ စနစ်ပါ။ ဥပမာ - "အိမ်ရှင်နိုင်ဖို့ 90% သေချာတယ်" လို့ ပြောပြီးမှ ရှုံးသွားရင် Log Loss က အမှတ်အများကြီး လျှော့ချပစ်ပါတယ်။

**ယှဉ်ပြိုင်ရမယ့် ပြိုင်ဘက်များ (Baselines):**
Model တစ်ခုက တကယ်ကောင်းမကောင်း သိဖို့ အောက်ပါ အခြေခံ ၃ ခုနဲ့ ယှဉ်ရပါတယ်။
1. **Base rates:** ဘာမှမသိဘဲ သမိုင်းတစ်လျှောက် နိုင်/သရေ/ရှုံး ရာခိုင်နှုန်းကို ရမ်းဖြေတဲ့ အဆင့် (Log Loss: 1.0528)
2. **ELO Rating:** စစ်တုရင်မှာ သုံးတဲ့ တွက်နည်းပါ။ နိုင်ရင် အမှတ်တိုး၊ ရှုံးရင် အမှတ်လျှော့တဲ့ စနစ် (Log Loss: **0.9529**). ဒါက **"မဖြစ်မနေ ကျော်ဖြတ်ရမယ့် အမှတ်အသား (The Bar)"** ပါ။
3. **Odds (ဒိုင်တွေရဲ့ ပေါက်ကြေး):** ဒါကတော့ ကမ္ဘာပေါ်မှာ အတော်ဆုံး လူသားတွေ၊ ငွေကြေးတွေ အများကြီးသုံးပြီး တွက်ထားတဲ့ ကြေးပါ။ (Log Loss: **0.9309**). ဒါက **"အမြင့်ဆုံး မျက်နှာကျက် (The Ceiling)"** ပါ။ ဘယ် Model မှ ဒါကို မကျော်နိုင်ပါဘူး။ ကျော်သွားရင် Code မှားနေလို့ (Leakage ဖြစ်နေလို့) သာ ဖြစ်ရပါမယ်။

---

### အပိုင်း (၃) - လက်တွေ့တည်ဆောက်ခြင်းနဲ့ ရှုံးနိမ့်မှုများ (The ML Begins)

**ပထမဆုံး ကြိုးစားမှု (LightGBM ဖြင့်):** Decision Tree လို့ခေါ်တဲ့ Model ကိုသုံးပြီး စမ်းပါတယ်။ ရလဒ်က ELO ကို မမီတဲ့အပြင် သာမန်ရမ်းဖြေတာလောက်ပဲ ရှိပါတယ်။ ဘာဖြစ်လို့လဲဆိုတော့ **"Overfitting"** (အလွတ်ကျက်မှတ်လွန်းခြင်း) ဖြစ်သွားလို့ပါ။ ဥပမာ - ကျောင်းသားက ဖော်မြူလာကို နားမလည်ဘဲ သင်္ချာပုစ္ဆာတွေကို အလွတ်ကျက်သွားတော့ စာမေးပွဲမှာ ဂဏန်းပြောင်းလိုက်တာနဲ့ တွက်မရတော့သလိုမျိုးပါ။

**ဖြေရှင်းနည်း (Early Stopping):** Model ကို အလွတ်ကျက်ခွင့် မပေးတော့ဘဲ လုံလောက်တဲ့ အနေအထားရောက်ရင် သင်ကြားမှုကို ရပ်ပစ်လိုက်ပါတယ်။ ရလဒ်ပြန်ကောင်းလာပေမယ့် ELO ကိုတော့ မနိုင်သေးပါဘူး။ Data နည်းနေတဲ့ အချိန်မှာ (ပွဲ ၄,၅၀၀ လောက်နဲ့) AI Model က ELO လို ရိုးရှင်းတဲ့ သင်္ချာပုံသေနည်းကို မယှဉ်နိုင်သေးတာ တွေ့ရပါတယ်။

**အောင်မြင်မှုဆီသို့ (The Blend - ပေါင်းစပ်ခြင်း):**
Model က ELO ကို မနိုင်တဲ့အခါ နှစ်ခုလုံးကို ပေါင်းလိုက်ပါတယ်။ ELO ရဲ့ ရလဒ် (68%) နဲ့ LightGBM ရဲ့ ရလဒ် (32%) ကို ပေါင်းပြီး **"Final Probability"** ထုတ်လိုက်တဲ့အခါ ပထမဆုံးအနေနဲ့ ELO ကို ကျော်ဖြတ်နိုင်သွားပါတယ်။ 

နောက်ထပ် အောင်မြင်မှုကတော့ **"Exponential Decay"** ကို သုံးလိုက်တာပါ။ လွန်ခဲ့တဲ့ ၁၀ လက ကစားခဲ့တဲ့ ပွဲရလဒ်ထက်စာရင်၊ ပြီးခဲ့တဲ့ အပတ်က ကစားခဲ့တဲ့ ရလဒ်ကို အလေးသာမှု (Weight) ပိုပေးလိုက်တာဖြစ်ပါတယ်။ (ဒါက လက်တွေ့ ဘောလုံးလောကရဲ့ Form အတက်အကျ သဘောတရားနဲ့ အတိအကျ ကိုက်ညီသွားပါတယ်)။

---

### အပိုင်း (၄) - စမ်းသပ်မှု (၁၀) ခု (The Age of Hypotheses)

ဒီအပိုင်းမှာ စိတ်ကူးသစ်တွေကို စမ်းသပ်ပါတယ်။ သင်ခန်းစာရစရာ အချက်တွေကတော့ -

* **Dixon-Coles Model (ရှုံးနိမ့်ခြင်း):** ဂိုးအရေအတွက်ကို ခန့်မှန်းတဲ့စနစ်ပါ။ ဒါပေမဲ့ COVID အချိန်က ကွင်းပိတ်ကစားရတဲ့အတွက် "အိမ်ကွင်းအားသာချက် (Home Advantage)" တွေ ကမောက်ကမဖြစ်ပြီး Model လည်း တိုင်ပတ်သွားပါတယ်။
* **သမိုင်းကြောင်း Data ထပ်ဖြည့်ခြင်း (အောင်မြင်ခြင်း):** EPL ရဲ့ နောက်ထပ် ၁၀ နှစ်စာ Data ကို ထပ်ထည့်လိုက်တဲ့အခါ Model က ပိုပြီး ဉာဏ်ကောင်းလာပါတယ်။ (Machine Learning မှာ Data များလေ ပိုကောင်းလေဆိုတာ သက်သေပြလိုက်တာပါပဲ)။
* **Championship (တန်းခွဲ) Data များ ထည့်ခြင်း (ရှုံးနိမ့်ခြင်း):** Data များတိုင်း ကောင်းတာမဟုတ်ပါဘူး။ ဒုတိယတန်းခွဲဖြစ်တဲ့ Championship ကစားဟန်က EPL နဲ့ မတူတဲ့အတွက် (သရေကျတာ ပိုများလို့) Model က ဇဝေဇဝါ ဖြစ်သွားပါတယ်။
* **Head-to-head (ထိပ်တိုက်တွေ့ဆုံမှု) (အကြီးမားဆုံး အောင်မြင်မှု):** တချို့အသင်းတွေက ခြေစွမ်းကောင်းပေမယ့် တချို့အသင်းတွေနဲ့တွေ့ရင် "ဂျိုထောက်" (Bogey team) ဖြစ်နေတတ်ပါတယ်။ ဒီလို အသင်းနှစ်သင်း ထိပ်တိုက်တွေ့ခဲ့တဲ့ ရလဒ်တွေကို Model ကို သင်ပေးလိုက်တဲ့အခါ အကြီးမားဆုံး တိုးတက်မှုကို ရရှိခဲ့ပြီး နောက်ဆုံး Champion Model အဖြစ် သတ်မှတ်နိုင်ခဲ့ပါတယ်။

---

### အပိုင်း (၅) - အဆုံးအဖြတ် စာမေးပွဲကြီး (The Final Exam)

အစောပိုင်းက သော့ခတ်သိမ်းထားတဲ့ 2024-2026 Data (ပွဲပေါင်း ၇၆၀) ကို ဖွင့်ပြီး Final Test စစ်လိုက်ပါတယ်။

ရလဒ်ကတော့ စိတ်မကောင်းစရာပါ။ **Model က ELO ကို မနိုင်တော့သလောက် ဖြစ်သွားပါတယ်။** (Validation မှာ 0.0041 သာခဲ့ပေမယ့် Test အစစ်မှာ 0.0003 ပဲ သာပါတော့တယ်)။ ဒါ ဘာကြောင့်လဲ?

ဒါကို **Validation Overfitting** လို့ ခေါ်ပါတယ်။ စမ်းသပ်မှုပေါင်း ၁၈ ခုလောက်ကို Validation Data ပေါ်မှာချည်းပဲ ထပ်ခါထပ်ခါ လုပ်ခဲ့တဲ့အတွက် Model က အဲဒီ ၇၆၀ ပွဲနဲ့ အံဝင်ခွင်ကျ ဖြစ်အောင် သူ့ဘာသာ မသိမသာ ပြောင်းလဲသွားတာပါ။ (ဒါပေမဲ့ ဒီစမ်းသပ်မှုက ရှုံးနိမ့်တာမဟုတ်ပါဘူး။ တကယ်လို့ Test Split ကိုသာ သော့မခတ်ထားခဲ့ရင်၊ အပြင်လောကမှာ တကယ်အလုပ်လုပ်မယ်လို့ ထင်ယောင်ထင်မှားဖြစ်ပြီး Production ပေါ်တင်လိုက်မိမှာပါ။ ခုတော့ အမှန်တရားကို အစောကြီး သိလိုက်ရပါတယ်)။

### အပိုင်း (၆) နဲ့ (၇) - Champion Model ဖွဲ့စည်းပုံနှင့် စနစ်လုံခြုံရေး

အနိုင်ရသွားတဲ့ `blend_v11` Model က (ELO Logistic စနစ် + LightGBM Decision Tree စနစ်) နှစ်ခုကို 48:52 အချိုးနဲ့ ပေါင်းစပ်ထားတာပါ။ 

ဒီ Model ကြီး အလိုအလျောက် အလုပ်လုပ်ဖို့အတွက် Code နဲ့ သတ်မှတ်ထားတဲ့ **Guardrails (အကာအကွယ် ဘောင်များ)** ထည့်ထားပါတယ်။ ဥပမာ - Model က ဒိုင်ပေါက်ကြေး (Odds) ထက် ပိုပြီး သိသာစွာ ခန့်မှန်းနိုင်နေရင်၊ မှန်လွန်းနေရင် "တစ်ခုခု မှားနေပြီ (Bug ရှိနေပြီ)" လို့ ယူဆပြီး အလိုအလျောက် ရပ်တန့်ပစ်မယ့်စနစ် မျိုးပါ။ Dashboard တွေ၊ Automation တွေနဲ့ API တွေချိတ်ပြီး ဆာဗာပေါ်မှာ သူ့ဘာသာသူ Run နေအောင် တည်ဆောက်ထားပါတယ်။

---

### အပိုင်း (၈) နဲ့ (၉) - Operations နဲ့ Betting Lab (လက်တွေ့ အသုံးချခြင်း)

ဒီစနစ်က နေ့စဉ် မနက်တိုင်း သူ့ဘာသာသူ ဒေတာအသစ်တွေ ဒေါင်းလုဒ်ဆွဲတယ်၊ ပွဲရလဒ်တွေ ထွက်လာရင် မှန်/မမှန် တိုက်စစ်တယ် (Settle လုပ်တယ်)၊ လိုအပ်ရင် Model ကို ပြန် Train ပါတယ်။ 

စိတ်ဝင်စားစရာ အကောင်းဆုံးကတော့ **Betting Lab** ပါ။ ဒီ Model ရဲ့ ရလဒ်တွေကို မြန်မာနိုင်ငံက ဘောလုံးပွဲလောင်းတဲ့ စနစ် (ဘော်ဒီ / ဂိုးပေါင်း) (Asian Handicap / Over-Under) အဖြစ် ပြောင်းလဲလိုက်တာပါ။ 
* Model က "ဘယ်သူနိုင်မလဲ" ကိုပဲ တိုက်ရိုက်တွက်တာမဟုတ်ဘဲ၊ ဖြစ်နိုင်ခြေရှိတဲ့ "ဂိုးအရေအတွက် (Scorelines)" တွေကို Dixon-Coles စနစ်နဲ့ ခန့်မှန်းပါတယ်။ 
* ပြီးမှ ဒိုင်ကပေးတဲ့ ပေါက်ကြေး (ဥပမာ - ၁ လုံးရှုံး၊ ပြား ၉၀) တွေနဲ့ ချိန်ထိုးပြီး၊ ဒီပွဲက လောင်းသင့်တဲ့ပွဲ (Value ရှိတဲ့ပွဲ) ဟုတ်မဟုတ် ဆုံးဖြတ်ပေးပါတယ်။
* ဒါပေမဲ့ တကယ့် ပိုက်ဆံနဲ့ မလောင်းခင် စက္ကူပေါ်မှာပဲ (Paper Betting) မှတ်ပြီး အရင်စမ်းသပ်တဲ့ သဘောပါ။

### အပိုင်း (၁၀) - နိဂုံး (The Scoreboard)

အနှစ်ချုပ်ရရင်တော့ ဒါက "AI ကို ဒေတာတွေ ပေးလိုက်တာနဲ့ ချက်ချင်း အဖြေမှန်တွေ ထွက်လာမယ်" ဆိုတဲ့ စိတ်ကူးယဉ်ဆန်တဲ့ ဇာတ်လမ်းမျိုး မဟုတ်ပါဘူး။ 

စမ်းသပ်မှုပေါင်းများစွာ လုပ်ခဲ့တယ်၊ အများစုက ကျရှုံးခဲ့တယ်၊ ကျရှုံးမှုတိုင်းကနေ နောက်ထပ် မလုပ်ရမယ့် အရာတွေကို သင်ယူနိုင်ခဲ့တယ်။ Data အသစ်တွေ၊ ပုံမှန်မဟုတ်တဲ့ Head-to-head လို Data တွေကသာ တကယ့် တိုးတက်မှုကို ပေးနိုင်တယ်ဆိုတာ တွေ့ခဲ့ရပါတယ်။ 

အခုဆိုရင် စနစ်က သူ့ဘာသာသူ ရပ်တည်နေပြီဖြစ်ပြီး၊ သူ့ရဲ့ အစစ်မှန်ဆုံး အကဲဖြတ်မှုကတော့ အနာဂတ်မှာ ကစားမယ့် "မကစားရသေးတဲ့ ဘောလုံးပွဲတွေ (Matches that have not happened yet)" ကို ဘယ်လောက် မှန်အောင် ခန့်မှန်းနိုင်မလဲ ဆိုတာပါပဲ။ 

အတိုပြောရရင်တော့ - **"Model တစ်ခုရဲ့ အရည်အသွေးဟာ၊ အဲဒီ Model ကို စစ်ဆေးတဲ့ စာမေးပွဲ (Harness) ရဲ့ တင်းကျပ်မှုပေါ်မှာပဲ မူတည်တယ်။ မှန်ကန်တဲ့ စာမေးပွဲစနစ်က ဘယ်သူ့ကိုမှ မျက်နှာမလိုက်ဘူး (does not flatter anyone)"** ဆိုတာကို ကောင်းကောင်းကြီး ပြသသွားပါတယ်။
